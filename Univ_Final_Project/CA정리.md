## 1. CA(Certificate Authority) 기관이란?

CA는 디지털 인증서를 발급하고 관리하는 신뢰할 수 있는 **민간 기업**이에요.
브라우저 회사들(구글, 애플, MS 등)이 신뢰를 보증하는 구조예요.

> 참고: [Microsoft Learn - X.509 인증서](https://learn.microsoft.com/ko-kr/azure/iot-hub/reference-x509-certificates)


### CA가 하는 일

```
1. 인증서 발급
   → 서버나 사용자의 신원 확인 후 인증서 발급

2. 인증서 서명
   → CA의 개인키로 인증서에 서명하여 신뢰성 보장

3. 인증서 폐지
   → 더 이상 신뢰할 수 없는 인증서를
     폐지 목록(CRL)에 등록

4. 신뢰 체계 유지
   → 최상위 루트 CA부터 하위 CA까지
     신뢰 체계 관리
```

### 대표적인 CA 기관

| 기관 | 나라 | 특징 |
|------|------|------|
| DigiCert | 미국 | 세계 최대 CA, 네이버/카카오/토스 등 사용 |
| Comodo | 영국 | 저렴한 인증서로 유명 |
| GlobalSign | 벨기에 | 유럽 최대 CA |
| Let's Encrypt | 미국 | 비영리, 완전 무료, 개발자들이 많이 사용 |
| 한국정보인증 (KICA) | 한국 | 국내 공공기관/정부 위주 |

### 왜 CA를 믿어야 하냐면

```
브라우저(크롬, 사파리, 엣지 등)에
신뢰할 수 있는 CA 목록이 미리 내장돼있어요

이 CA들이 발급한 인증서면
브라우저가 자동으로 신뢰해요
→ 주소창에 자물쇠  표시

우리가 만든 모의 CA는 목록에 없어서
→ 브라우저 경고  표시
```

## 2. X.509 인증서 구조

X.509는 인증서의 국제 표준 형식이에요. (RFC 5280 기반)

인증서에는
서버 이름, 유효기간, 공개키, CA 서명이 있음
```

### 인증서 안에 들어있는 것 (v3 기준)

| 필드 | 설명 | 예시 |
|------|------|------|
| 버전 (Version) | 인증서 버전 번호 | V3 |
| 일련번호 (Serial Number) | CA가 발급한 고유 번호 | 123456789 |
| 발급자 (Issuer) | 인증서를 발급한 CA 이름 | CN=DigiCert |
| 유효기간 (Validity) | 시작일 ~ 만료일 | 2026-01-01 ~ 2027-01-01 |
| 주체 (Subject) | 인증서 소유자 이름 | CN=naver.com |
| 공개키 (Public Key) | 암호화에 사용하는 키 | RSA 2048bit |
| 서명 알고리즘 | CA가 서명에 사용한 알고리즘 | SHA256withRSA |
| CA 서명 (Signature) | CA가 도장 찍은 것 | (암호화된 값) |

### 공개키 vs 개인키

```
자물쇠와 열쇠로 이해하면 쉬움

공개키 = 자물쇠
→ 누구한테나 나눠줘도 됨
→ 이걸로 데이터를 잠금 (암호화)

개인키 = 열쇠
→ 서버만 가지고 있음
→ 이걸로만 열 수 있음 (복호화)

즉
→ 누군가 공개키로 데이터 잠그면
→ 개인키 가진 서버만 열 수 있음
→ 중간에 가로채도 못 읽음
```

### 인증서 파일 형식

| 형식 | 설명 |
|------|------|
| .crt / .pem | Base64로 인코딩된 인증서. `-----BEGIN CERTIFICATE-----`로 시작 |
| .key | 개인키 파일 (절대 외부 노출 금지) |
| .csr | 인증서 서명 요청 파일 (CA한테 발급 신청할 때 사용) |
| .p12 / .pfx | 개인키 + 인증서 합친 파일 (Spring Boot에서 사용) |

---

```

## 3. HTTPS 통신 흐름

### HTTP vs HTTPS

```
HTTP (보안 없음)
→ 데이터가 그냥 날아감
→ 중간에 누가 가로채면 다 보임
→ 카페 와이파이에서 해킹 가능

HTTPS (보안 있음)
→ 데이터가 암호화돼서 날아감
→ 중간에 가로채도 못 읽음
→ 주소창에 자물쇠  표시
```

### 실제 중간자 공격 예시

```
HTTP일 때
사용자가 결제 버튼 클릭
→ paymentKey=tviva2026...
→ amount=48000
→ secretKey=test_sk_xxx
이 데이터가 그냥 날아감, 해킹 당할 수 있음.

HTTPS일 때
사용자가 결제 버튼 클릭
→ x7Ks9mP2qL8nR4vT1wZ6... (암호화)
```

### HTTPS 연결 흐름

```
1. 브라우저가 서버 접속 요청

2. 서버가 인증서 제출

3. 브라우저가 CA 목록에서 검증

4. 유효기간 확인

5. 공개키로 암호화 통신 시작
                                    
```

### 인증서 발급 흐름

```
기업(네이버)이 CA(DigiCert)한테 신청
→ "저 네이버인데 인증서 주세요"
→ DigiCert가 "진짜 네이버 맞는지" 확인
→ 확인되면 인증서 발급 (유효기간 1년)
→ 1년 동안 그냥 씀
→ 만료되기 전에 갱신

사용자가 접속할 때마다 발급하는 게 아니라 이미 발급받은 인증서를 계속 쓰는 것.

```
